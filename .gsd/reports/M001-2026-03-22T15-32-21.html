Knowledge 30

Rules 4

IDScopeRule
#ScopeRule
1acquisitionKeep raw evidence at the card-fragment level by default (`overlay_title`, visible subtitle/price texts, canonical URL, image alt) rather than archiving full pages.
2product-uiFuture dashboard work must default to a French, broad-audience product surface that separates market overview, full listing exploration, listing detail, and runtime administration instead of one compact technical page.
3operationsNever promote a synchronized SQLite file straight into `data/vinted-radar.db`; create a remote snapshot, copy to a temporary local path, run DB health checks, then rename atomically.

Patterns 12

IDPattern
#Pattern
1The public `/catalog` HTML embeds the full Homme/Femme catalog tree inside `self.__next_f.push(...)`, and listing cards are already server-rendered under `div.new-item-box__container`.
2Keep per-run listing history in `listing_observations` and retain `listing_discoveries` as the lower-level per-sighting debug layer.
3Public item pages expose `can_buy`, `is_closed`, `is_hidden`, and `is_reserved` inside escaped script text; normalizing escaped quotes before matching keeps item probes browserless.
4Keep premium scoring demand-led and only apply a contextual price boost when the selected peer group clears explicit support thresholds.
5Drive the dashboard HTML and JSON diagnostics from the same server-side payload assembly so the browser view stays debuggable and truthful.
6Persist operator runtime state in `runtime_cycles` and let both the CLI and dashboard read that table instead of inventing separate scheduler state files.
7Treat catalog-first acquisition as the preferred scaling posture and use listing detail pages only as a targeted fallback, not as the default per-item path.
8Treat the primary image URL timestamp as `published_at_estimated`, never as an exact publication timestamp.
9Treat Cloudflare / Turnstile challenge rate as a first-class acquisition signal rather than assuming that a successful live run means the current HTTP path scales cleanly.
10Use `https://www.vinted.fr/api/v2/catalog/items` as the primary catalog discovery surface, and keep public item pages as a separate targeted evidence path for state resolution.
11Keep the market-summary dashboard and the broad listing explorer as separate surfaces; the explorer should page directly from SQL instead of loading the full scored corpus just to browse listings.

Lessons 14

IDLesson
#What Happened
1Live discovery needed both broad seed coverage and low request volume for smoke verification.
2Live CLI output can fail on real titles even when the data pipeline is correct.
3Narrow fixture-only probe parsing left real 200 item pages unresolved.
4Premium ranking becomes noisy fast when price context is too sparse.
5A first-pass live dashboard can look repetitive even when the UI is correct.
6The live dashboard can truthfully show a `running` runtime cycle with zeroed latest-run scan counts while still displaying useful rankings and history.
7Healthy long-running runtime logs were being flagged as process errors by the tool harness.
8A limited `continuous` run can look like broad multi-day evidence when it is actually revisiting the same small subset of catalogs.
9User feedback made clear that the M001 dashboard reads like an internal proof surface rather than a usable intelligence workspace.
10A server-side catalog or item fetch that works in one run can degrade into widespread 403 / Turnstile challenges a few hours later.
11The primary image URL timestamp is useful but not exact.
12Older SQLite proof databases can become unreadable if schema bootstrap creates indexes for columns that are only added later by migration.
13Multi-day operator metadata is not enough for milestone sign-off if listing-history tables are unhealthy.
